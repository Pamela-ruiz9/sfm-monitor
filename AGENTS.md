## What this project is

Public dashboard of risk indicators for the Mexican Financial System (SFM) — IMOR / IMORA / ICOR / IFRS9 stages, FX, policy rate, TIIE, INPC. Authored by **Ingrid Pamela Ruiz Puga** (data scientist, BBVA México) with **Artemio Padilla** as co-author of the 2026 blueprint and Astro migration.

Live: `pamela-ruiz9.github.io/sfm-monitor` · License: code MIT, content CC-BY 4.0.

## Hard rules (do not violate)

1. **Don't break the live dashboard.** Any change to root `index.html` or `data/sfm-data.json` ships immediately to production via GitHub Pages. If unsure, work in `app/` instead.
2. **Don't push or tag without explicit user confirmation.** Local commits and tags are fine; `git push` is not. (Exception: branch `feat/app-redesign-pwa-v0.2.0-dev` was pushed and PR #1 opened with explicit user request.)
3. **Don't modify `data/sfm-data.json` by hand.** It's regenerated by `.github/workflows/update-data.yml` (Banxico daily) and a manual CNBV monthly process. Direct edits get clobbered.
4. **Don't re-introduce the deleted `sfm-risk-monitor` repo direction.** That bifurcation was abandoned; everything lives here in `sfm-monitor`.
5. **Inside `app/`: never import `'../../data/sfm-data.json'` directly from a component.** Always go through `src/data/loader.ts` so Zod validation and caching apply.
6. **Inside `app/`: never use `any` or `as any`.** TS is strict with `noUncheckedIndexedAccess` + `exactOptionalPropertyTypes`. If you reach for `any`, the schema in `src/data/schema.ts` is wrong — fix the schema instead.
7. **Branded types in financial APIs.** New helpers operating on percentages, basis points, or currency must use the branded types in `app/src/types/branded.ts`. No raw `number` for financial quantities crossing module boundaries.
8. **Never claim Lighthouse / visual / chart-rendering success without actually loading the page in a browser.** `npm run build` proves SSR + type checks; it does not prove the chart renders correctly. Say so explicitly.

## Common tasks

### "Migrate chart X from legacy to Astro"
1. Read the chart's logic in `index.html` (search for its container ID).
2. Find/extend the matching shape in `app/src/data/schema.ts` — replace any `z.unknown()` or `z.looseObject` with the real shape.
3. Run `cd app && npm run build` — Zod will complain at runtime if the shape is wrong with real data.
4. Create `app/src/components/charts/<Name>Chart.tsx` following the pattern in `FXChart.tsx` (Chart.js as island).
5. Add to `app/src/pages/index.astro` with `client:visible`.
6. Update `app/README.md` migration table + `docs/migration-astro.md` checklist.

### "Add a new indicator"
1. Update CNBV/Banxico extraction script (likely needs a new commit to whatever processes raw data into `data/sfm-data.json`).
2. Add field to Zod schema in `app/src/data/schema.ts`.
3. Bump `data/sfm-data.json` schema-version in CHANGELOG (MINOR if additive).
4. Build chart following the migration pattern above.

### "Bump a dependency"
- `app/`: `cd app && npm update <pkg>` then `npm run build` to verify.
- Root `index.html` uses CDN-hosted Chart.js with pinned versions in `<script>` tags — bump cautiously, the legacy code is fragile.

### "Fix the build"
- `cd app && npm run build` failing → likely Zod schema doesn't match real `data/sfm-data.json`. Inspect the failing path with `node -e "console.log(JSON.stringify(require('./data/sfm-data.json').<path>, null, 2))"` and adjust schema. Do **not** loosen the schema globally; tighten it per-field.

### "Set up Zenodo DOI" or "release v0.1.0"
- Read `docs/citability.md` end-to-end. Most steps require manual user action via Zenodo web UI; no agent should attempt them automatically.

## When you're done

1. Run `npm run build` in `app/` if you touched anything there.
2. Update `CHANGELOG.md` `[Sin publicar]` section with what changed.
3. If you added a chart: update `app/README.md` migration table.
4. If you added a doc: link it from `README.md` or another doc — orphans hurt discoverability.
5. Commit with the project's prefix style: `feat(app):`, `docs(citability):`, `chore:`, `fix:`. End trailers with `Co-Authored-By: <model>`.
6. **Do not push or create PRs without explicit user instruction.**

## What NOT to do

- ❌ Refactor the legacy `index.html` — it's a tombstone, not living code.
- ❌ Add features to the legacy stack — they all go to Astro app now.
- ❌ Use Tailwind v3 patterns (`tailwind.config.js`) — `app/` uses v4 with `@theme` in `global.css`.
- ❌ Add `pnpm-lock.yaml` or `yarn.lock` — the repo is npm-only.
- ❌ Suggest migrating to Vercel / Next.js / Remix — the blueprint already evaluated and chose Astro + Cloudflare Pages.
- ❌ Re-create the deleted `sfm-risk-monitor` repo direction. That bifurcation was retired.
- ❌ Use emojis in code or commit messages unless the user explicitly asks.
- ❌ Touch `LICENSE` or `LICENSE-CONTENT` — dual licensing is intentional and legally meaningful.
